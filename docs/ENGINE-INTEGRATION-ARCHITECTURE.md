# SowyVid — Engine Integration Architecture

How SowyVid consumes the Jorge Engine Vault engines. One engine is integrated per
phase (see `docs/ENGINE-VAULT-CATALOG.md`); the rest stay `DEFERRED`.

## The adapter sandwich (mandatory for every engine)

```
SowyVid application model
        ↓  input adapter  (src/features|adapters/<domain>/…)
Generic engine input
        ↓  engine (packages/<engine>/)
Generic engine result
        ↓  output adapter
SowyVid application / renderer
```

The application may depend on an engine. **An engine must never depend on**
SowyVid, React, Electron, SQLite, UI, product branding, or app-specific paths —
unless it explicitly needs a platform transport (e.g. BridgeDrop's LAN server),
and then that transport is isolated behind an interface.

## Import strategy per engine

Allowed strategies: workspace package · local dependency · **vendor copy with
preserved package boundary** · compiled consumption · selective port.

**Chosen for Northstar (and the default for this vault): vendor copy with
preserved package boundary.**

- The engine lives under `packages/<engine>/` with its **own** `package.json`,
  `tsconfig*.json`, `README`, and tests — the package identity is preserved.
- SowyVid consumes it through a path **alias to its `src`** (e.g.
  `@jorge-engines/northstar-creative` → `packages/northstar-creative-engine/src/index.ts`),
  compiled and tested by SowyVid's own toolchain.
- The engine's original tests run inside SowyVid's `vitest` (added to `include`).

### Why not npm workspaces / separate build?

The vault packages pin **TypeScript 7 preview / vitest 3/4** dev toolchains. In
this environment `npm install` for those workspaces **fails** (`Exit handler
never called!`). Adding `"workspaces"` to the app root and running each engine's
own `tsc` build would couple SowyVid's green build to that failing install and to
`vitest` major-version skew. The vendor-copy strategy keeps:

- the package boundary (engines stay generic, in `packages/`, unmodified in spirit),
- a single, reliable build/test toolchain (SowyVid's TS 5.7 + vitest 2.1),
- the engines' own tests running and passing under that toolchain.

The engines' `dist/` and `node_modules/` are not vendored; the source is the
source of truth and the app builds it. `packages/**` is excluded from the app's
ESLint (third-party conventions) but **is** typechecked and tested.

## Where adapters live

```
src/
  features/creative/       Northstar adapters + service (Phase A, done)
  adapters/media/          MediaVault adapters      (Phase B, deferred)
  adapters/visual/         FrameLogic adapters      (Phase C, deferred)
  adapters/audio/          SoundWeave adapters      (Phase D, deferred)
  adapters/phone/          BridgeDrop adapters      (Phase E, deferred)
  adapters/ai/             PromptGate adapters      (Phase F, deferred)
```

(Northstar's adapters currently live under `src/features/creative/`; later engines
follow the `src/adapters/<domain>/` convention. Both are acceptable narrow
boundaries.)

## Per-phase integration checklist (from the vault directive)

1. Check the catalog. 2. Confirm the engine matches the phase. 3. Re-read its
README/API/schemas/tests. 4. Validate against current architecture. 5. Narrow
integration plan. 6. Add SowyVid adapters. 7. Integrate only that engine.
8. Add integration tests. 9. Update the catalog. 10. Commit separately.

## Versioning & reproducibility (Section 8)

Each integrated engine persists its identity with generated data. For Northstar,
a project's `creative` block stores `engineVersion`, `family`, `variantId`,
`conceptId`, `seed`, `inputFingerprint`, `targetDurationSec`. Future engines add
`visualEngineName/Version`, `audioEngineName/Version`, etc. Existing persisted
data is never silently regenerated by a newer engine; upgrades require a migration
or an explicit regeneration action.

## Source vault protection

The ZIP `Jorge-Engine-Vault-v1.0.0.zip` is never modified. A legitimate engine bug
is fixed in the vendored copy under `packages/`, documented, and reported as a
candidate for a future Engine Vault v1.0.1 — never silently changed in the source.
